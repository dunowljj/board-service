# ADR-0001: Architecture Decision Records 도입

## Status
Accepted

## Date
2026-03-01

## Context

프로젝트가 성장하면서 기술적·구조적 결정들이 코드에 묵시적으로 녹아들고,
나중에 합류하는 팀원이나 리뷰어는 "왜 이렇게 설계했는가"를 추적할 수 없게 된다.

결정 근거가 문서화되지 않으면:
- 동일한 토론이 반복된다.
- 변경 시 기존 결정이 의도치 않게 훼손된다.
- AI 협업 환경에서 Claude가 승인 없이 방향을 추가로 결정하는 리스크가 생긴다.

이 프로젝트는 AI(Claude)를 실행 파트너로 사용하므로,
결정 경계를 명시적으로 관리하는 메커니즘이 특히 중요하다.

## Decision

Architecture Decision Records(ADR) 형식을 채택한다.

- 저장 위치: `docs/adr/`
- 인덱스: `docs/adr/README.md`에서 관리
- 상태 값: `Proposed` | `Accepted` | `Superseded`
- Superseded ADR은 히스토리 보존을 위해 삭제하지 않는다.
- ADR 번호는 4자리 순번으로 부여한다 (예: 0001, 0002, …).

### ADR 작성 기준
다음 조건 중 하나라도 해당하면 ADR이 필요하다:
- 시스템 경계, 불변식, 또는 기본 동작을 정의하거나 변경하는 경우
- 성능, 일관성, 신뢰성 트레이드오프를 도입하는 경우
- 리뷰어가 "왜 이렇게 설계했는가"라고 물을 수 있는 경우

다음은 ADR 불필요:
- 단순 리팩터링
- 버전 범프
- 경계에 영향 없는 설정 변경

## Consequences

**긍정적 영향**
- 결정 근거가 코드와 함께 버전 관리된다.
- 동일 주제에 대한 반복 토론이 줄어든다.
- Claude가 승인된 범위 밖에서 결정을 내리는 것을 구조적으로 방지한다.
- 새 팀원의 온보딩 비용이 감소한다.

**부정적 영향 / 트레이드오프**
- ADR 작성·유지에 추가 비용이 발생한다.
- 경계 판단이 명확하지 않을 때 오버헤드가 생길 수 있다.
